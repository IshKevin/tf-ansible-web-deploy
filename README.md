# Infrastructure Deployment & Configuration Project

##  Executive Summary

This project demonstrates a complete **DevOps automation workflow** that provisions and configures a web application environment on AWS.

Using:
- **Terraform (v1.5+)** → Infrastructure provisioning  
- **Ansible (v2.14+)** → Configuration management  

The system deploys an **EC2 instance** and configures it with an **Nginx web server**, following a structured workflow:

> **Bootstrap → Remote Backend → Infrastructure → Configuration → Verification → Teardown**

This implementation emphasizes:
- Infrastructure as Code (IaC)
- Automation and reproducibility
- Secure access management
- Real-world DevOps practices

---

# Architecture & Design Rationale

##  1. Bootstrap Layer (Backend Initialization)

Terraform is first executed using **local state** to create backend infrastructure:

- **S3 Bucket** → stores Terraform state  
- **DynamoDB Table** → enables state locking  

###  Why this approach?

Terraform cannot use a remote backend that does not yet exist. Therefore:

1. Backend resources are created using local state  
2. Terraform is then reconfigured to use the remote backend  
3. State is migrated to S3  

---

##  2. Remote Backend Configuration

After bootstrap, Terraform is configured to use:

- **S3** → state storage  
- **DynamoDB** → state locking  

###  Benefits

- Centralized state management  
- Prevents concurrent modification issues  
- Enables state versioning and recovery  

###  State Separation

Two independent state files are maintained:

```

bootstrap/terraform.tfstate
infra/terraform.tfstate

````

✔ Same backend infrastructure  
✔ Different state keys (best practice)

---

## 3. Infrastructure Layer (Terraform)

Terraform provisions the application infrastructure.

### EC2 Instance
- Type: `t2.micro`  
- OS: Amazon Linux 2  
- Reason:
  - Cost-effective (Free Tier)
  - Optimized for AWS
  - Compatible with Ansible

---

### Security Group

Configured to allow:

- **SSH (22)** → administrative access  
- **HTTP (80)** → public web traffic  

---

### SSH Key Management

Terraform dynamically generates SSH credentials:

- `tls_private_key` → generates RSA key pair  
- Public key → registered in AWS  
- Private key → stored locally  

```bash
chmod 400 web-key.pem
````

 Ensures secure, password-less SSH access for Ansible

---

##  4. Configuration Layer (Ansible)

Ansible configures the EC2 instance after provisioning.

---

###  Automated Inventory (Terraform Integration)

The Ansible inventory file is **automatically generated by Terraform** using:

```hcl
resource "local_file" "ansible_inventory" {
  content = templatefile("${path.module}/inventory.tftpl", {
    ip_address = aws_instance.web.public_ip,
    key_path   = "../terraform/infra/${aws_key_pair.generated.key_name}.pem"
  })
  filename = "../../ansible/inventory.ini"
}
```

###  Why this is important

* Eliminates manual configuration
* Prevents IP mismatches
* Maintains Terraform as the **single source of truth**
* Enables a fully automated pipeline

---

### Role-Based Configuration

Ansible uses a modular **role (`webserver`)**.

#### Responsibilities:

* Enable Nginx repository (`amazon-linux-extras`)
* Install Nginx
* Start and enable service
* Deploy web application (`index.html`)
* Restart service via handlers when needed

✔ Modular
✔ Reusable
✔ Industry-standard approach

#  Deployment Workflow

##  Step 1: Bootstrap Backend (Local State)

```bash
cd terraform/bootstrap
terraform init
terraform apply
```

---

##  Step 2: Configure Remote Backend

```bash
terraform init
```

Confirm state migration when prompted

---

##  Step 3: Provision Infrastructure

```bash
cd ../infra
terraform init
terraform apply -auto-approve | tee ../../APPLY.txt
```

 Infrastructure is created
 Ansible inventory is generated automatically

---

##  Step 4: Configure Server (Ansible)

```bash
cd ../../ansible
ansible-playbook -i inventory.ini site.yml
```

---

##  Step 5: Verification

###  CLI Test

```bash
curl http://<public-ip>
```

###  Browser Test

Open:

```
http://<public-ip>
```

# Teardown & Cleanup

##  Destroy Infrastructure


